# 天津市市政工程局综合管理系统架构设计

## 一、系统概述

**系统名称**: 天津市市政工程局综合管理系统

**技术栈**: Python + MySQL + Tkinter

**系统定位**: 面向天津市市政工程局机关及下属单位的综合管理平台，不同处室拥有不同的功能权限

**核心处室功能**:
- 资产管理处：资产登记、分配、管理、统计（全系统资产）
- 办公室：公文审核、消息通知(局名义)、业务请示处理
- 劳动人事处：账号注册、密码修改审批
- 科技处：科研项目管理、科研经费审批
- 财务处：预算审批、预算分配
- 其他处室：查看本部门资产、提交申请、查看消息
- 下属单位：单位资产、公文提交、同级函件、科研申报、预算申报

## 二、组织架构

### 2.1 组织架构层级说明

本系统采用**三级管理架构**：

| 层级 | 名称 | 说明 |
|------|------|------|
| 第一级 | 局机关 | 19个机关处室 + 局领导 |
| 第二级 | 中层单位 | 20个下属单位（负责管理基层单位，其中10个设有基层单位） |
| 第三级 | 基层单位 | 各中层单位下属的基层业务单位 |

## 五、功能模块设计

### 5.1 局领导功能
- 综合仪表盘：查看全局概览
- 分管处室交互：查看分管处室工作动态（根据分管范围）
- 待办审批：重要事项审批
- 消息接收：接收重要通知
- 统计分析：查看关键指标
> 说明：党委书记(L1)具有最高权限，可与所有处室和下属单位交互 | 党委副书记、局长(L2)负责行政口处室 | 党委副书记、副局长(L3)负责党政人事口处室 | 党委副书记(L4)协助党政人事口

### 5.2 资产管理处功能
- 资产登记：新增资产信息
- 资产管理：修改、删除、资产调拨
- 资产查询：按类别、状态、组织查询
- 资产统计：数量统计、金额统计、处室分布
- 申请审批：审批资产使用申请

### 5.3 办公室功能
- 公文起草：创建局名义公文
- 公文审核：审核各处室/下属单位提交的公文
- 公文发布：发布审核通过的局名义公文
- 局通知发布：向所有处室/下属单位发送局名义通知
- 下属单位请示处理：接收并回复下属单位的业务请示
- 收发记录：查看所有公文和通知收发记录

### 5.4 计划处功能
- 项目计划管理：编制年度项目计划
- 立项审批：审核项目立项申请
- 投资管理：跟踪投资项目进展
- 统计报表：项目投资统计

### 5.5 设施管理处功能
- 设施台账管理：设施信息录入维护
- 维护计划：制定设施维护计划
- 设施查询：查询设施分布及状态

### 密码修改流程（申请审批制）

```
用户提交申请 → password_change_applications表 → 劳动人事处审批 → 密码生效
```

1. 用户在"个人中心"提交密码修改申请
2. 申请存储在 password_change_applications 表（状态为 pending）
3. 劳动人事处(user with organization_id=11)审批申请
4. 审批通过后更新 users 表密码，状态改为 approved
5. 审批拒绝则状态改为 rejected

## 八、核心流程

### 8.1 登录流程
1. 输入用户名密码
2. 验证并获取用户角色和组织
3. 根据角色分发到对应视图

### 8.2 密码修改流程（申请审批制）
1. 用户在"个人中心"填写原密码、新密码、确认密码
2. 系统验证原密码正确性及新密码一致性
3. 申请提交到 password_change_applications 表（状态：pending）
4. 劳动人事处审批申请（批准/拒绝）
5. 审批通过后更新 users 表密码

### 8.3 资产管理流程(资产管理处)
资产登记 → 资产分配 → 资产变更 → 资产统计

### 8.4 公文流程
- **局名义公文**: 起草 → 办公室审核 → 局领导签发 → 办公室发布 → 各单位接收
- **处室公文**: 起草 → 处室负责人审核 → 发送给目标单位
- **下属单位请示**: 起草 → 提交 → 相关处室审核 → 办公室备案 → 回复

### 8.4 消息通知流程
- **局名义通知**: 办公室编写 → 发布 → 各处室/下属单位接收
- **处室通知**: 各处室编写 → 发送给目标单位
- **业务请示**: 下属单位向相关处室发送 → 处室回复

### 8.5 科研项目流程(科技处)
下属单位申报 → 科技处审核 → 批准立项 → 经费拨付 → 进度跟踪 → 验收

### 8.6 预算管理流程(财务处)
局机关预算申请 → 财务处审批
下属单位预算申请 → 财务处审批 → 指标分配 → 执行跟踪

## 十二、REST API 架构

### 12.1 架构概述

系统提供 REST API 接口层，基于 FastAPI 框架实现，将原有业务服务（Service）封装为 HTTP 接口，支持：
- 接口自动化测试
- 第三方系统集成
- 移动端后端服务

### 12.2 技术栈

| 组件 | 技术 | 说明 |
|-----|------|------|
| Web 框架 | FastAPI | 现代高性能 Web 框架 |
| API 文档 | Swagger UI | 自动生成可交互 API 文档 |
| 数据校验 | Pydantic | Python 类型提示和数据校验 |
| ASGI 服务器 | Uvicorn | 高性能 ASGI 服务器 |

### 12.3 目录结构

```
tianjin_bureau_system/
├── api/                      # API 层（新增）
│   ├── __init__.py
│   ├── main.py              # FastAPI 应用入口
│   ├── auth.py              # 认证接口
│   ├── users.py             # 用户管理接口
│   ├── organizations.py     # 组织管理接口
│   ├── assets.py            # 资产管理接口
│   ├── budgets.py           # 预算管理接口
│   ├── documents.py         # 公文管理接口
│   └── messages.py          # 消息管理接口
├── services/                 # 业务逻辑层（原有）
│   ├── auth_service.py
│   ├── asset_service.py

│   └── ...
├── models/                   # 数据模型层（原有）
```

### 12.4 设计原则

1. **复用业务逻辑**：API 层只做 HTTP 封装，调用原有的 Service 层
2. **统一响应格式**：`{"code": 200, "message": "...", "data": {...}}`
3. **RESTful 风格**：使用标准的 HTTP 方法和 URL 路径
4. **自动文档**：通过 FastAPI 自动生成 Swagger 文档

### 12.5 响应格式

成功响应：
```json
{
    "code": 200,
    "message": "操作成功",
    "data": {...}
}
```

错误响应：
```json
{
    "code": 400,
    "detail": "错误信息"
}
```

### 12.6 启动方式

```bash
# 方式1：直接运行
python -m uvicorn api.main:app --reload --port 8000

# 方式2：指定地址
python -m uvicorn api.main:app --host 0.0.0.0 --port 8000
```
